## 1. Context & goal

When the player dies, the run ends. v1.9 adds an optional **revive**: on the game-over screen (Free mode only), a "đź“ş Watch & Continue" button plays a rewarded ad and, on completion, resumes the *same* run â€” score, combo, and altitude preserved â€” instead of restarting. The revive re-enters play safely by granting the existing jetpack **boost** (invulnerable upward flight for `boostDuration`), so the player clears whatever killed them and flies back into the platform field. The ad itself is abstracted behind the bridge: a web **stub** ships now; a real network (native AdMob) plugs in later without touching game code.

## 2. Principles preserved

- **Pure, deterministic sim.** The new `revive(world)` lives in `src/game/sim.ts`, is DOM-free, and is a deterministic function of world state â€” unit-tested. It reuses existing physics (`boostT`); no new movement rules.
- **Daily fairness intact.** Revive is **Free-mode only**. Daily mode (the deterministic, seeded, replayable competitive leaderboard) never revives â€” so its determinism and fairness are byte-identical to today. Determinism + reachability tests stay green.
- **Reachability.** Revive places the player mid-view among the already-spawned (reachable-by-construction) platforms and grants a boost; it never fabricates unreachable state.
- **Ads isolated + offline-first.** The ad is behind an optional bridge method + a web stub; if neither the native ad nor the stub resolves a reward, the game simply stays on the normal game-over screen. No ad ever blocks play. Anti-cheat is unchanged (Firestore rules; score cap 100000; monotonic).
- **Budgets:** core < 60KB gzip; no image/audio assets (procedural stub); no new runtime deps. UI copy minimal + TR/EN.

## 3. Architecture

## 4. Components

### 4.1 `revive(world)` (sim, pure, deterministic)

```
export function revive(world: World): void {
  world.over = false;
  const p = world.player;
  p.y = world.cameraY + world.viewHeight * 0.5; // respawn in the middle of the view (above the death line)
  p.prevY = p.y;                                 // collapse interpolation so the render doesn't streak
  p.vy = 0;
  p.boostT = TUNING.boostDuration;               // invulnerable upward flight â€” clears hazards, reaches platforms
  world.events = [];                             // don't replay 'gameover'
  world.stompFx = [];
}
```

Deterministic (a pure function of the passed world). Reuses the boost the pickup already grants, so the re-entry is safe by construction (during boost the player plows through enemies and flies up into the reachable platform field, then resumes normal physics when the boost ends). Score-bearing state (`maxAltitude`, `stompBonus`, `combo`, `maxCombo`) is untouched, so the run continues where it left off.

### 4.2 Ad bridge + web stub

`PlatformBridge` gains `showRewardedAd?(): Promise<boolean>`. The proxy `bridge` exposes it as an optional passthrough (like `showLeaderboard`). Native (future) implements it with AdMob rewarded.

`src/ui/ad.ts` `showRewardedAdStub(root: HTMLElement, seconds = 3): Promise<boolean>` renders a `.modal-overlay .ad-overlay` "đź“ş {adPlaceholder}" card with a live countdown; when the countdown reaches 0 it removes the overlay and resolves `true` (reward earned). It is a rewarded-ad placeholder (no skip). Because it is a `.modal-overlay`, it inherits the v1.6.x `pointer-events:auto` fix.

### 4.3 Game-over UI (`screens.ts`)

`showGameOver(..., onRevive?: (() => void) | null)`: when `onRevive` is provided, render a prominent primary button "đź“ş {watchContinue}" (first, above "Again") that calls `onRevive`. When it is null/absent (daily mode, or the revive already used), the game-over screen is exactly as today. `showGameOver` still calls `clearCombo()` first (v1.6.x) and hides the pause button.

### 4.4 `main.ts` orchestration

- Add `let revivesUsed = 0;` reset to 0 in `onPlay`.
- At game-over, compute `canRevive = mode === 'free' && revivesUsed < 1;`. Pass `onRevive = canRevive ? doRevive : null` into both `showGameOver` calls (instant + rank re-render).
- `doRevive` (async): show the ad â€” `const rewarded = bridge.showRewardedAd ? await bridge.showRewardedAd() : await showRewardedAdStub(uiRoot);` â€” and only on `rewarded === true`: `revivesUsed++; revive(world); playing = true; loop.start(); ui.showHud();` (resume the same run; `revive` already cleared `world.over`; `showHud` hides the game-over panel + shows the HUD/pause button). On `false`, do nothing (stay on the game-over screen; the button remains available).
- Guard against a stale revive after the player moved on (mirror the existing `overGen` pattern): capture the game-over generation and ignore the revive if a new run started meanwhile.

The score already submitted at game-over is monotonic (Firestore + `best = max`), so re-finalising at the next (higher) game-over after a revive is correct and idempotent.

## 5. Determinism & fairness

- `revive` is deterministic; the pure-sim tests cover it. Daily mode never calls it â†’ daily seeded replay is byte-identical to today.
- Free-mode revive extends a casual/global run; the score is legitimately achieved (the player did climb that high, with a revive) and remains under the anti-cheat cap. No pay-to-win in the *fair* (daily) board.

## 6. Testing

- **`revive(world)` unit tests** (`test/sim.test.ts` additions): after `world.over = true` then `revive`, `over` is false, `player.boostT === TUNING.boostDuration`, the player's `y` is above the death line (`cameraY + viewHeight`), and score-bearing state (`maxAltitude`, `stompBonus`, `combo`) is unchanged; `events` is emptied. A "reachable after revive" sanity step: stepping the world post-revive does not immediately set `over` again.
- **Determinism/reachability suites:** unchanged and green (daily replay identical; revive is Free-only and not exercised by the daily determinism tests).
- **Ad stub + game-over button + resume:** verified by `pnpm build` + a live Brave/Playwright smoke â€” die in Free mode, the "Watch & Continue" button shows; tap it â†’ the stub ad plays â†’ the run resumes (HUD back, player boosting, score preserved); in Daily mode the button never shows; a second death offers no further revive. ## 7. Acceptance criteria

1. Core < 60KB gzip; all existing tests green unmodified + new revive tests green; determinism & reachability intact; Daily replay byte-identical.
2. Dying in **Free** mode shows a "đź“ş Watch & Continue" button on game-over; **Daily** mode never shows it.
