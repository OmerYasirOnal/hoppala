# Hoppala v1.3 — Control Feel + Altitude Zones + Zone Progression

- **Status:** Approved (brainstorm 2026-07-05; user pre-approved full build)
- **Spec round:** v1.3 (game-feel + retention progression)
- **Predecessors:** v1.2 online foundation (`docs/specs/2026-07-04-hoppala-v1.2-online-engagement-design.md`) — this round reuses its online layer for cloud sync.
- **North star:** retention first (users return), ads later — every piece is weighed by return-to-play impact. See memory `hoppala-monetization-strategy`.

## 1. Context & goal

Hoppala v1.2 shipped online leaderboard + cloud save. v1.3 makes the *climb itself* feel better and goal-driven, to increase session length and day-N return:

1. **Control feel** — smoother steering + tunable sensitivity + a settings slider (the current control teleports the player straight to the finger target, no smoothing/sensitivity).
2. **Altitude zones** — named, colored, themed altitude bands with a HUD zone indicator (extends the existing altitude-driven sky gradient into discrete named zones).
3. **Persistent zone progression** — the highest zone ever reached is tracked and cloud-synced; first-reach triggers a celebration; a Zones screen shows the collection ("4/8"). This is the retention hook.
4. **Clearer steer guidance** on first run.

Constraint that binds everything: the deterministic core stays deterministic and offline-first; the online layer (v1.2) is reused, never required.

## 2. Principles preserved

- **Pure simulation:** `sim.ts`/`spawner.ts`/`rng.ts`/`daily.ts` stay DOM-free and deterministic. New zone logic (`src/game/zones.ts`) is also pure and DOM-free.
- **Reachability invariant** (vertical gap ≤ `0.8 × JUMP_HEIGHT`) — untouched; smoothing is horizontal only.
- **Determinism:** the steering *smoothing* is added inside the sim but is a pure function of `(playerX, targetX)` — daily replays stay consistent within v1.3. Steering *sensitivity* lives in the INPUT layer (`drag.ts`) and never reaches the sim, so it cannot affect determinism or daily fairness. Zones/celebration are pure functions of `maxAltitude` computed in the composition/render layer, never in the sim.
- **Budgets:** core entry < 60KB gzip (CI-enforced by `scripts/check-bundle-size.mjs`); 60fps.
- **UI copy minimal, TR/EN** via `navigator.language` + the v1.2 `setLang` override; strings in `src/ui/screens.ts`. Code/docs English.
- **Online layer reuse:** cloud sync of `maxZone` extends the v1.2 `CloudSave`/orchestrator; no new backend. Firebase stays a lazy chunk. Anti-cheat unaffected (zone is derived from score, which rules already cap).

## 3. Non-goals (v1.3 boundary)

- No new game mechanics (enemies, mini-games) — those are a separate sub-project.
- No coins/currency; zones unlock by climbing, not purchase.
- No ads (future round).
- No per-zone gameplay differences (physics identical across zones) — zones are visual + progression only in v1.3. (Zone-specific hazards could be a later round.)
- Sensitivity is user-tunable; **smoothing strength is a fixed tuned constant** (not user-facing) to keep the settings surface minimal.

## 4. Control feel

### 4.1 Sensitivity (input layer — `src/input/drag.ts`)

`attachDrag` multiplies the finger delta by a sensitivity factor before moving the target:

```
target += (e.clientX - lastClientX) * scale() * sensitivity()
```

- `sensitivity()` is a getter passed into `attachDrag` (default returns the persisted setting). Range **0.5–2.0**, default **1.0** (1.0 = today's 1:1 feel; the slider lets players tune to taste).
- Lives entirely in the input layer — the sim still receives an absolute `targetX`, so **determinism/daily fairness are unaffected**.

### 4.2 Smoothing (sim layer — `src/game/sim.ts`)

Replace the teleport `p.x = wrapX(targetX)` with a wrap-aware ease toward the target:

```
p.x = wrapX(p.x + shortestWrapDelta(p.x, targetX) * TUNING.steerSmoothing)
```

- `shortestWrapDelta(from, to)` returns the signed shortest horizontal delta across the 0..viewWidth wrap seam (range `(-w/2, w/2]`).
- `TUNING.steerSmoothing ∈ (0, 1]`: `1` = today's instant snap; **default `0.5`** (tuned for "smooth but still responsive" — must remain precise enough to land platforms; final value validated live). A new `TUNING` constant.
- Pure and deterministic → daily replays stay identical within v1.3 (they differ from v1.2 daily runs, which is expected for a new version).
- `shortestWrapDelta` is a small pure helper (in `sim.ts` or `zones.ts`'s sibling), unit-tested for the wrap seam.

### 4.3 Settings slider

Add a **"Hassasiyet / Sensitivity"** range input (0.5–2.0, step 0.1) to the settings screen (`src/ui/settings.ts`), below the language row. Persisted via a new `Save.sensitivity` field. `main.ts` passes `() => loadSave().sensitivity ?? 1` into `attachDrag`. Changing it takes effect immediately (no reload needed — the getter is read per drag).

## 5. Altitude zones

### 5.1 Zone model (pure — `src/game/zones.ts`)

DOM-free module, unit-tested. Exports:

```ts
export interface Zone {
  key: string;                 // stable id, e.g. 'clouds'
  meters: number;              // lower boundary in meters (score units)
  top: [number, number, number];    // gradient top rgb
  bottom: [number, number, number]; // gradient bottom rgb
}
export const ZONES: Zone[];              // ordered, ascending meters, ZONES[0].meters === 0
export function zoneIndexAt(meters: number): number;      // highest zone whose boundary ≤ meters
export function zoneProgress(meters: number): number;     // 0..1 within the current zone toward the next boundary (1 in the last zone)
```

Zone set (8 zones; boundary in meters = score; altitude px = meters × 10). Exponential-ish spacing: early zones close (quick wins), later far (aspirational):

| # | key | TR / EN name | boundary (m) |
|---|-----|--------------|--------------|
| 0 | meadow | Çayır / Meadow | 0 |
| 1 | sky | Gökyüzü / Sky | 100 |
| 2 | clouds | Bulutlar / Clouds | 250 |
| 3 | dawn | Şafak / Dawn | 450 |
| 4 | aurora | Kutup / Aurora | 700 |
| 5 | strato | Stratosfer / Stratosphere | 1100 |
| 6 | space | Uzay / Space | 1700 |
| 7 | cosmos | Kozmos / Cosmos | 2500 |

The last zone (`cosmos`) is endless (`zoneProgress` returns 1). Exact RGB palettes are set in `ZONES` and validated live (screenshot the real game); the design fixes the *hue direction* per the §Bölüm-2 table, the plan fixes concrete values.

Zone **names** are localized: `src/ui/screens.ts` `STRINGS` gains `zone_<key>` entries (tr/en). `zones.ts` stays name-free (pure); the UI maps `key → localized name`.

### 5.2 Rendering (`src/render/renderer.ts`)

Replace the ad-hoc `SKY` array with `ZONES`-driven gradient: the background lerps between the current zone's palette and the next zone's palette by `zoneProgress` (same lerp machinery already present). Star density ramps in the upper zones (reuse existing). No hard color jumps — transitions stay smooth.

### 5.3 HUD zone indicator (`src/ui/screens.ts`)

During play, the HUD shows: the current **zone name** next to the meters, plus a **thin progress bar** to the next zone boundary (width = `zoneProgress`). Minimal, matches the existing HUD styling. `main.ts` updates it each frame from `meters` (throttled like the score: only on change of zone/bar-bucket to avoid layout thrash).

### 5.4 Zone-entry banner

When the player crosses into a higher zone *during a run* (current zone index increases), a brief center banner shows the zone name (e.g. "Bulutlar!") for ~1.2s, then fades. This is the in-run feedback; the *first-ever* reach additionally triggers the persistent celebration (§6). Reuses the coach-tip overlay style.

## 6. Persistent zone progression

### 6.1 Data

- `Save` gains `maxZone?: number` (highest zone index ever reached; default 0). `CloudSave` gains `maxZone: number`; `mergeCloudSave` takes the **max**. Local `toCloudSave`/`writeCloudSave` map it.
- On each step where `zoneIndexAt(meters) > storedMaxZone`, `main.ts`: updates `maxZone`, persists locally (`saveMaxZone`), pushes to cloud best-effort (a new `online.pushMaxZone(z)` mirroring `pushBest`, or fold into the existing cloud-save patch), and fires the **first-reach celebration**.

### 6.2 First-reach celebration

A celebration overlay: "Yeni bölge: Aurora! 🎉" with the zone color, shown once when a brand-new zone is reached (not on subsequent runs through it). Dismissible / auto-fades. Distinct from the lightweight §5.4 banner (which shows every time you cross a boundary in a run).

### 6.3 Zones screen (the collection)

A new overlay (`src/ui/zones.ts`, mirroring `leaderboard.ts`/`settings.ts` modal pattern), reachable from the menu (a "🗺️ Bölgeler / Zones" button) and/or the profile area: lists all 8 zones; reached ones rendered lit with their color + localized name + boundary; locked ones dimmed with a lock glyph; a header "Ulaşılan: X/8". This is the retention "collection" surface. Read-only; no online call needed (uses local `maxZone`, which is cloud-merged on init).

## 7. Steer guidance

Enhance the existing first-run onboarding tip (v1.2 `showOnboardingTip`) to be clearer and more prominent about swiping: a larger, animated "👆 Sağa/sola sürükle / Swipe to steer" hint on the first play, fading after ~2.5s. Gated by the existing `Save.onboarded` flag (no new state). Keep it a single non-blocking overlay.

## 9. Testing

Pure, unit-tested (Vitest): `zoneIndexAt` (boundaries, below 0, above last), `zoneProgress` (0..1, last-zone=1), `shortestWrapDelta` (across the seam, exact half-width), sensitivity mapping, smoothing ease (converges toward target, wrap-aware), `mergeCloudSave` maxZone (max), storage roundtrip of new fields. UI (settings slider, zones screen, HUD indicator, banners) verified by `pnpm build` + a live smoke (screenshot the real zones/gradient) per the codebase pattern. Reachability + all existing tests stay green.

## 10. Acceptance criteria

1. Core entry < 60KB gzip (CI). All existing tests + reachability green.
2. Sensitivity slider changes steering responsiveness immediately; persists; does not affect the sim (daily replays identical for a fixed seed regardless of sensitivity).
3. Steering is visibly smoother (eased, not teleporting) yet still precise enough to land platforms (validated live).
4. Background renders named zones with smooth transitions; HUD shows current zone + progress bar; crossing a boundary shows the banner.
5. Reaching a brand-new highest zone shows the celebration once and persists (`maxZone`), survives reload, and syncs to cloud (visible after clearing local + reloading while signed in).
6. Zones screen shows reached/locked zones and the X/8 count.
7. Determinism: for a fixed daily seed, the run is byte-identical across sensitivity settings; smoothing is deterministic.

## 11. Roadmap context (later rounds)

Mini-games (enemies etc.) — separate sub-project. Zone-specific hazards/mechanics — possible later. Ads (rewarded revive tied to zones, interstitials) — the monetization endgame, its own round once retention is proven.
