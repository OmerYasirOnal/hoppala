# Hoppala v1.10 — Sky Poster menu, coherent screen flow, honest ads

Status: approved (brainstorm 2026-07-10). Ships as web v1.10.0 and iOS 1.10.0.

## 1. Why

Three coupled problems, reported by the user:

1. **Ads show nothing.** In the shipped iOS build (`TESTING=false`, real ad unit), tapping "Watch & Continue"
   does nothing — no ad appears. Root cause: the AdMob app is brand-new and unverified ("İnceleme gerekli"),
   so the real rewarded unit returns **no-fill**; `showRewardedAd()` resolves `false` and `doRevive` silently
   aborts. The integration code is correct — there is simply no real ad inventory until the app is approved/live.
   Secondarily, when no ad is available the button **fails silently**, which is bad UX and an App-Review risk.
2. **No real menu screen.** The "menu" is a bare `<h1>Hoppala</h1>` + hint + three stacked pill buttons floating
   on the raw canvas. No screen design, no identity.
3. **Incoherent flow.** Screen swaps are instant `.hidden` toggles (jarring pops), and — the key gap — the
   **game-over screen has no path back to the main menu**. After dying you can only replay / view leaderboard /
   share; the only route to the menu is Pause → Main-menu mid-run.

## 2. Goals / non-goals

**Goals:** a proper "Sky Poster" entry menu; a coherent, animated flow between menu ↔ HUD ↔ game-over with a
game-over → main-menu path; ad-readiness gating so the revive button never dead-taps; a verification path that
lets the user actually see the rewarded ad play on-device.

**Non-goals (unchanged):** no `src/game/*` or `src/core/*` change — sim, spawner, RNG, determinism, the
reachability invariant and daily fairness stay **byte-identical**. No new ad formats (rewarded only). No
gameplay/tuning change. The Settings / Leaderboard / Journey / Nickname modals keep their current card style
(already coherent) — only the menu and game-over screens are redesigned, adopting that same card language.

## 3. Binding constraints (from CLAUDE.md)

- Pure simulation untouched; all work is in UI adapters (`src/ui/*`), styles, `src/main.ts` composition, and the
  `src/platform/*` bridge. No DOM imports leak into sim files.
- Core bundle stays < 60KB gzip (this is pure CSS/DOM — negligible). AdMob stays in the lazy Capacitor chunk.
- UI copy in the minimal `STRINGS` set (`src/ui/screens.ts`), auto TR/EN. Code/docs English.
- Conventional commits. `pnpm test` + `pnpm build` stay green.
- Transitions must respect `prefers-reduced-motion` and hold 60fps (CSS-only, no JS animation loop).

## 4. Design

### 4.1 Sky Poster visual system

The menu and game-over render into the existing `#panel`, restyled as a **poster card**: a centered,
semi-transparent rounded card (same palette/shape as `.modal`: `#121a2c`-ish, `border:1px solid #22304d`,
rounded 20px, soft shadow) at ~78% opacity so the live sky gradient + parallax clouds (already drawn by the
canvas/v1.7 fx) show through behind it. Card `pointer-events:auto`; backdrop stays transparent so the game
remains visible.

**Menu contents (top→bottom):**
- **Mascot**: a small doodle matching the in-game player — a yellow (`#ffd23e`) round body with two dark
  (`#1c1c28`) eyes — as inline SVG, with a gentle CSS bounce (`translateY`) loop and an occasional blink.
- **Wordmark** "Hoppala": large, bold, warm, subtle letter-spacing + drop shadow (playful, not a logo asset).
- **Primary** button **Başla / Play** (filled green `#7ee08a`, large).
- **Secondary** button **Günlük #N / Daily #N** (ghost/outline).
- **Icon row**: 🏆 (Leaderboard) · 🗺️ (Journey) · ⚙ (Settings) as three compact circular icon buttons in one
  row (replaces the three stacked full-width pills).
- **Stats chip**: a subtle single line — `Rekor: N m` and, when available, the global rank (as today).

**Game-over contents (same card language):**
- Big score, then the existing record / daily-best / best / rank / best-combo sub-lines.
- Buttons, in this order:
  - **📺 İzle & Devam** — shown **only when a rewarded ad is actually ready** (see 4.3).
  - **Tekrar / Again** — primary (replays the same mode).
  - **🏠 Ana menü / Main menu** — the new path back to the menu (uses the existing `mainMenu` string).
  - Icon row: 🏆 (Leaderboard) · Paylaş/Share.

### 4.2 Transitions

A small isolated helper in `screens.ts` swaps panel content with a fade+scale (opacity 0→1, scale .96→1,
~180–220ms via CSS transition/keyframes). Game-over settles in with a brief backdrop vignette; the menu has a
gentle entrance. All effects are CSS keyframes/transitions gated behind a `.anim` class; a
`@media (prefers-reduced-motion: reduce)` block disables them. No changes to the rAF game loop.

Because `#ui` sets `pointer-events:none` for gameplay pass-through, every interactive layer must re-assert
`pointer-events:auto` (existing lesson — the poster card and icon buttons must set it, like `.modal-overlay`).

### 4.3 Honest ads (readiness gating)

- **Bridge** (`src/platform/bridge.ts`): add optional `isRewardedAdReady?(): boolean` and
  `preloadRewardedAd?(): void` to `PlatformBridge`, exposed through the `bridge` delegator the same
  getter-guarded way as `showRewardedAd` (absent on web).
- **Native** (`src/platform/admob.ts`): export `isRewardedAdReady()` returning the internal `loaded` flag, and
  `preloadRewardedAd()` wrapping the existing `preload()`. (`initAdMob` already preloads; each show re-preloads.)
- **Web** (`src/main.ts` fallback): when `bridge.showRewardedAd` is undefined we are on web → the stub is always
  available, so readiness = `true`.
- **Gating decision** — a small **pure, unit-tested** function:
  ```
  shouldOfferRevive(mode, revivesUsed, hasNativeAd, nativeAdReady): boolean
    = mode === 'free' && revivesUsed < 1 && (hasNativeAd ? nativeAdReady : true)
  ```
  `main.ts` uses it to decide whether to pass `onRevive` to `showGameOver`.
- **Late-fill reveal**: on game-over in free mode with `revivesUsed < 1`, if native and not yet ready, call
  `preloadRewardedAd()` and set one bounded re-check (~2.5s, cleared/guarded by the current `overGen`); if the
  ad becomes ready in that window, re-render game-over with the revive button. On genuine no-fill the button
  simply never appears — the game is fully playable without it (App-Review-safe).

### 4.4 Versioning

Web in-app version display and iOS marketing version → **1.10.0** (new App Store train, so no "train closed"
issue). Native build number: **4** for the `TESTING=true` verification build, **5** for the `TESTING=false`
submission build.

## 5. Files

- `src/ui/screens.ts` — new `showMenu` (poster + mascot + icon row) and `showGameOver` (add Main-menu button,
  ad-ready gating via `onRevive`), the transition helper, new `createUI` handler `onMainMenu`.
- `src/styles.css` — poster card, wordmark, mascot + bounce/blink keyframes, icon-row buttons, screen
  transitions, game-over vignette, `prefers-reduced-motion`.
- `src/main.ts` — `shouldOfferRevive` wiring + late-fill preload/re-check; `onMainMenu` handler (→ `showMenu`);
  version string `1.10.0`.
- `src/platform/bridge.ts` — `isRewardedAdReady?` + `preloadRewardedAd?` on the interface and delegator.
- `src/platform/admob.ts` — export `isRewardedAdReady()` + `preloadRewardedAd()`.
- `src/platform/capacitor.ts` — wire the two new methods into `nativeBridge`.
- `src/ui/screens.ts` STRINGS — reuse existing `mainMenu`; the menu sub-line reuses the existing `hint`
  string, so no new tagline key is added.
- `src/ui/revive-policy.ts` (NEW) — the pure exported `shouldOfferRevive(...)` helper, with a Vitest unit
  test (`src/ui/revive-policy.test.ts`). Isolated in its own file so the test imports it without pulling in
  `main.ts`'s import-time side effects.
- iOS: `ios/App/App.xcodeproj/project.pbxproj` marketing 1.10.0 + build number; `admob-config.ts` `TESTING` flag.

## 6. Testing

- **Unit (Vitest):** `shouldOfferRevive(...)` truth table — free+ready→true, free+not-ready→false, free+web→true,
  daily→false, revivesUsed≥1→false. Keeps the review-safe gating logic covered deterministically.
- **Build:** `pnpm test` + `pnpm build` green; core payload < 60KB gzip; no `src/game`/`src/core` diff.
- **Visual QA (Playwright/Brave, dev build):** menu look; menu→play→game-over→menu transitions; game-over
  Main-menu button returns to menu; revive button present on web (stub always ready); reduced-motion path.
- **On-device (TestFlight, build 4, `TESTING=true`):** the user verifies the menu, transitions, and a real
  rewarded **test ad** playing through the İzle & Devam → revive → resume flow.

## 7. Ship runbook

1. Build the feature on `feat/v1.10-menu-flow-ads`; adversarial review; fix wave; visual QA.
2. PR → independent zero-context review → merge to `master` → Vercel prod deploy (web menu/flow live; ad code
   inert on web, stub unchanged).
3. Flip `TESTING=true`; bump iOS marketing 1.10.0 / build **4**; `pnpm build && npx cap sync ios`;
   archive + upload → TestFlight. **User device-tests** menu + transitions + revive-with-test-ad.
4. On user approval: flip `TESTING=false`; build **5**; archive + upload.
5. ASC: attach build 5; **App Privacy → add Identifiers/Device ID used for advertising + Used-for-Tracking:
   Yes**; confirm the ATT string; Add for Review → user Submits.
6. (User's own step) AdMob payment/tax setup for payouts; the AdMob app-store link + app-ads.txt verification
   completes once the store listing is crawled.
