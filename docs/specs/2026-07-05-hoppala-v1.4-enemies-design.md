# Hoppala v1.4 — In-Climb Enemies (Stomp)

- **Status:** Approved (brainstorm 2026-07-05; user pre-approved spec → plan → workflow build)
- **Spec round:** v1.4 (first in-climb enemy — a new-content / variety retention round)
- **Predecessors:** v1.2 online, v1.3 zones. Reuses the deterministic sim/spawner + the v1.3 zone tint.
- **North star:** retention first (variety keeps players climbing), ads later. See memory `hoppala-monetization-strategy`.

## 1. Context & goal

Hoppala is a pure endless vertical jumper. v1.4 adds **monsters inside the climb** (Doodle-Jump style) for variety and challenge, chosen over separate bonus-round mini-games to preserve the game's identity and the deterministic core. The interaction is **stomp**: land on a monster's head to kill it and bounce; any other contact kills you (single-life, consistent with falling). An active **jetpack boost plows through** monsters (synergy with existing pickups). Enemies spawn seeded (daily-compatible) and scale with altitude.

This is one focused feature: a single enemy type + the stomp/boost/death mechanic + seeded spawning + procedural rendering. Additional enemy types, per-zone themes, combos/score-pops, and a shoot alternative are explicitly deferred (§3).

## 2. Principles preserved

- **Pure simulation:** enemies live in `sim.ts`/`spawner.ts`/`types.ts` (DOM-free, deterministic, seeded `mulberry32`). Rendering/audio/haptics are thin adapters.
- **Determinism:** enemy spawn draws from the same seeded RNG stream, so runs stay reproducible (`run() === run()`). Daily mode works unchanged (seeded). NOTE: adding enemy RNG draws shifts the stream, so v1.4 daily layouts differ from v1.3 — expected for a new version; determinism *within* v1.4 holds.
- **Reachability invariant** (vertical platform gap ≤ `0.8 × JUMP_HEIGHT`) — UNTOUCHED. Enemies are **additive hazards** placed in gaps (like phantoms); they never replace or block the guaranteed reachable platform chain. A monster on/near a path is always navigable: steer around it, or stomp it (a stomp is itself a valid landing + bounce). Tests keep enforcing the invariant.
- **Budgets:** core entry < 60KB gzip (CI-enforced); 60fps. No image assets — the monster is procedural (consistent with the game's altitude-atmosphere visuals).
- **World is y-down**; `SimEvent`-driven audio/haptics.

## 3. Non-goals (v1.4 boundary)

- **One enemy type only** (a floating monster). Multiple types / per-zone themes → later rounds.
- **No shoot mechanic** (stomp only; no new input — steering stays "drag only").
- **No combo counter / score-pop** on stomps in v1.4 (the bounce is the reward). Deferred.
- **No interruption / separate mini-game** (rejected in brainstorm — enemies are in-climb).
- No enemy projectiles, no multi-hit health, no invulnerability frames beyond the existing jetpack boost.
- No new online/leaderboard surface (score is still meters; enemies don't add a score channel in v1.4).

## 4. Enemy model (`src/game/types.ts`)

```ts
export interface Enemy {
  id: number;
  x: number;      // center x (updated each step for patrol)
  y: number;      // center y (world y-down)
  baseX: number;  // patrol center
  amp: number;    // horizontal patrol amplitude; 0 = static
  speed: number;  // rad/s
  phase: number;
  dead: boolean;  // killed by stomp/boost — culled next step
}
```

`World` gains `enemies: Enemy[]` (initialized `[]` in `createWorld`, filled by the spawner, culled below the view like platforms/pickups).

`SimEvent` gains `'stomp'` (union: `'bounce' | 'spring' | 'break' | 'boost' | 'stomp' | 'gameover'`).

`TUNING` gains:
- `enemyR: 15` — collision + render radius.
- `stompVy: -1300` — bounce impulse on a stomp (stronger than `bounceVy` −1050, weaker than a spring's 1.8×; a satisfying reward). Derived-feel constant, tuned live.

## 5. Spawning (`src/game/spawner.ts`)

`Difficulty` gains `enemyRatio: number`. In `difficultyAt(altitude)` (altitude in px):

```ts
const enemyRatio = a < 1000 ? 0 : Math.min(0.15, ((a - 1000) / 8000) * 0.15);
```

Enemies begin ~100m up (after the player has learned the basics) and ramp to a 15% per-platform chance. `spawnExtras` returns an added `enemy: Enemy | null`, drawn AFTER the phantom + pickup checks (so their RNG order is unchanged within a call). An enemy spawns in the GAP above the main platform (like a phantom — never on the guaranteed landing spot), at a random x, static or horizontally patrolling:

```ts
let enemy: Enemy | null = null;
if (d.enemyRatio > 0 && rng() < d.enemyRatio) {
  const lastGap = prevY - mainPlat.y;
  const y = prevY - lastGap * (0.4 + rng() * 0.3);      // mid-gap, above the chain platform
  const patrol = rng() < 0.5;
  const amp = patrol ? 40 + rng() * 50 : 0;
  const usable = TUNING.viewWidth - 2 * (amp + TUNING.enemyR);
  const baseX = amp + TUNING.enemyR + rng() * Math.max(0, usable);
  enemy = { id: enemyId, x: baseX, y, baseX, amp, speed: patrol ? d.moveSpeed * (0.5 + rng() * 0.7) : 0, phase: rng() * Math.PI * 2, dead: false };
}
```

`spawnExtras` signature gains an `enemyId: number`; `sim.ts` `fillPlatforms` passes `world.idSeq++` for it and pushes any returned enemy to `world.enemies`.

## 6. Collision & interaction (`src/game/sim.ts`)

Each `step`, after the existing platform collision + boost handling, resolve enemies (patrol first, then contact):

1. **Patrol:** for each live enemy with `amp > 0`, `enemy.x = baseX + Math.sin(time * speed + phase) * amp` (mirrors moving platforms).
2. **Contact** (wrap-aware circle overlap between the player (radius `playerR`) and each live enemy (radius `enemyR`)):
   - **Boost active** (`player.boostT > 0`): kill the enemy (`dead = true`), emit `'stomp'`. No bounce needed (boost keeps driving), no death — the player plows through.
   - **Stomp** (not boosting, `player.vy > 0` i.e. falling, and the player's foot swept onto the enemy from ABOVE this step — same `footBefore <= enemyTop && footAfter >= enemyTop` test used for platforms, with `enemyTop = enemy.y - enemyR`): kill the enemy, set `player.vy = TUNING.stompVy`, snap `player.y = enemyTop - playerR`, emit `'stomp'`.
   - **Otherwise** (overlapping while rising, or side/below contact while not boosting and not a valid top-stomp): `world.over = true`, emit `'gameover'`.
3. **Cull** dead enemies and enemies below the view (same cull line as platforms/pickups): `world.enemies = world.enemies.filter((e) => !e.dead && e.y <= cullBelow)`.

Interaction with platform collision: the stomp check runs in the same falling window; if the player would both stomp an enemy and land on a platform in one sweep, the topmost contact wins (an enemy stomp and a platform land are both "land on the highest thing crossed" — resolve to the smaller `y`). The plan specifies the exact ordering; the guarantee is: a valid top-contact never kills the player, and enemies never block reaching a platform.

## 7. Rendering (`src/render/renderer.ts`)

Draw live enemies (procedural, no assets): a rounded body + two eyes + a hostile mouth/brow, tinted by the current zone's palette (reuse the zone color already computed for the sky) so monsters feel native to each zone. Placed at `enemy.x`/`enemy.y − camY`, radius `enemyR`, drawn with the same wrap-x triplet the player uses (so patrolling monsters wrap). Dead enemies are already culled, so nothing to draw for them. A brief scale/opacity pop on stomp is a nice-to-have (may be added in the plan if cheap).

## 8. Events / audio / haptics

- `SimEvent` `'stomp'` is emitted on kill (stomp or boost-plow).
- `src/audio/sfx.ts`: add a `'stomp'` FX (a short punchy tone, distinct from `'spring'`/`'boost'`).
- `src/platform/native-install.ts` `hapticFor`: map `'stomp'` → a medium impact (like `'spring'`/`'boost'`). The `HapticKind` record + the sfx FX record are exhaustive `Record` types keyed by event, so both gain a `'stomp'` entry (tsc enforces this).
- `main.ts` already fans `world.events` to `sfx.play` + `bridge.onEvent`; `'stomp'` flows through unchanged.

## 9. Testing

Pure, unit-tested (Vitest), no DOM:
- **Stomp:** falling player whose foot sweeps an enemy's top → enemy `dead`, `player.vy === TUNING.stompVy`, `'stomp'` event, no `gameover`.
- **Death:** player rising into (or side-overlapping, not boosting) an enemy → `world.over`, `'gameover'`, enemy NOT killed.
- **Boost plow:** player with `boostT > 0` overlapping an enemy → enemy `dead`, `'stomp'`, no `gameover`, still boosting.
- **Patrol:** a patrolling enemy's x oscillates deterministically around `baseX`.
- **Seeded determinism:** `run() === run()` with enemies in the stream (extend the existing determinism test to also compare `enemies.length`).
- **Reachability preserved:** the existing "keeps the board reachable while climbing" test still passes (enemies don't widen platform gaps).
- **Spawner:** `difficultyAt` `enemyRatio` is 0 below the threshold and monotonic-ish up to the cap; `spawnExtras` returns an enemy only within probability and never on the main platform's landing spot.

UI/render (the monster drawing) is verified by `pnpm build` + a live Brave smoke (screenshot a monster + a stomp) per the codebase pattern.

## 10. Non-functionals

- Determinism preserved; daily mode unaffected (seeded). Reachability invariant intact. Core entry < 60KB gzip (CI). All existing tests stay green (enemy RNG shift changes layouts, not test assertions, which check invariants/self-consistency, not exact positions).
- No i18n (enemies are visual). No online/backend change. No new dependencies.

## 11. Acceptance criteria

1. Core < 60KB gzip; all existing + new tests green; reachability invariant holds.
2. Monsters appear in the climb from ~100m, scaling with altitude, seeded (a fixed daily seed yields the same monsters).
3. Stomping a monster (land on its head) kills it and bounces the player up (`stompVy`); wrong contact (rising into / side, not boosting) ends the run.
4. An active jetpack boost plows through monsters (kill, no death).
5. Monsters render procedurally, zone-tinted, and patrol/wrap correctly; a stomp plays the `'stomp'` sfx + haptic.
6. Determinism: a fixed daily seed replays byte-identically.

## 12. Roadmap (later rounds)

More enemy types (per-zone themed behaviors), a stomp combo + score-pop (retention feedback), optional shoot mechanic as an alternate control, enemy projectiles/harder variants high up. Ads (rewarded revive after a monster death) — the monetization endgame, its own round.
