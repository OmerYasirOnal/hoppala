# Hoppala v1.2 — Online Foundation + Retention Loop (Phase 1)

- **Status:** Approved (brainstorm 2026-07-04)
- **Spec round:** v1.2 (new round; amends two v1 principles — see §2)
- **Predecessors:** `docs/specs/2026-07-02-hoppala-design.md`, `docs/specs/2026-07-02-v1.1-feature-pack-design.md`, `docs/specs/2026-07-02-ios-appstore-design.md`
- **This doc covers Phase 1 only.** Phases 2–3 are sketched in §14 for context and get their own spec rounds.

## 1. Context & goal

Hoppala is live (`hoppala.vercel.app`, PWA, already offline-capable) and in Apple App Review with Game Center leaderboards. The owner wants the game to be **user-friendly** and **retention-driven** ("players want to come back again and again"), **fully playable offline but enriched when online**, using **Firebase** as the backend — "managed well."

Phase 1 delivers the online foundation and the single strongest retention hook (competition) plus baseline UX friendliness:

1. A Firebase online layer (anonymous identity + nickname, offline queue/sync) that is **strictly additive** and **isolated** from the deterministic core.
2. A **unified cross-platform online leaderboard** (global + daily) — the "beat everyone" loop.
3. **Cloud save** of local progress under the anonymous UID.
4. Baseline UX: leaderboard screen, game-over rank, settings screen, nickname flow, minimal onboarding.

The overriding constraint: **the game must never depend on the network to be played.** Online is a lazy-loaded, gracefully-degrading enrichment.

## 2. Principles preserved — and the two amendments

Preserved (bind all changes):

- **Pure simulation.** `src/game/sim.ts`, `src/game/spawner.ts`, `src/core/rng.ts`, `src/core/daily.ts` stay DOM-free, deterministic, seeded, unit-tested. The online layer never touches them.
- **World model, TUNING, reachability invariant** — untouched. Existing tests stay green.
- **UI copy minimal**, TR/EN auto via `navigator.language`, strings in `src/ui/screens.ts`. Code/docs in English.
- **Conventional commits; CI runs `pnpm test` + `pnpm build`.**

Amended by this spec round (explicitly, because the owner opened online as a new round):

- **"Zero runtime dependencies"** → refined to **"zero runtime dependencies in the CORE game bundle."** Firebase is an *isolated, lazy-loaded* dependency that never enters the core entry chunk. CLAUDE.md is updated to say this.
- **"Online leaderboard / accounts" were v1 non-goals** → this round approves an online leaderboard and anonymous identity. Coins/cosmetics, real accounts, and analytics remain out of Phase 1 (see §3).

## 3. Non-goals (Phase 1 boundary)

Deferred to later phases or out of scope:

- Achievements, missions/goals, daily streak, reminder notifications → **Phase 2**.
- Cosmetic unlocks (characters/themes), friends, Google/Apple account linking, cross-device transfer → **Phase 3**.
- **No coin/currency economy, ever** (unlocks in Phase 3 gate on milestones, not currency).
- No analytics, no ads, no background music.
- No Cloud Functions and no deterministic replay validation in Phase 1 (see §7 — hardening path if cheating becomes real).
- iOS App Store update is a **follow-on task**, not a Phase-1 blocker (see §12).

## 4. Architecture

### 4.1 Layer separation

```
Deterministic core (untouched)         Adapters / composition            New online layer (this spec)
──────────────────────────────         ─────────────────────            ────────────────────────────
sim.ts spawner.ts rng.ts daily.ts  ←   main.ts (composition)      →      src/online/seam.ts   (stable no-op delegate)
types.ts (TUNING)                      storage.ts (local Save)           src/online/queue.ts  (PURE, localStorage)
                                       ui/screens.ts (DOM)               src/online/orchestrator.ts (glue)
                                       platform/bridge.ts (native)       src/online/firebase.ts (LAZY chunk, Firebase SDK)
                                                                         src/online/types.ts (OnlineBackend, ScoreRow…)
                                                                         src/online/nickname.ts (PURE)
                                                                         src/online/rank.ts (PURE format)
```

The online layer mirrors the existing **native-bridge seam pattern** (`src/platform/bridge.ts`): a stable delegate object with a no-op default, and a real implementation installed lazily at runtime. `main.ts` depends only on the stable seam, never on Firebase directly.

### 4.2 Module responsibilities

- **`src/online/types.ts`** — interfaces only. `OnlineBackend` (the Firebase-agnostic contract), `ScoreRow`, `RankInfo`, `BoardId`, `CloudSave`.
- **`src/online/seam.ts`** — stable `online` object delegating to an installed orchestrator; default no-op. Analogous to `bridge`. `main.ts` imports this.
- **`src/online/queue.ts`** — **PURE** localStorage-backed pending-submission queue. `enqueue`, `collapse` (keep max score per board), `drain`. No Firebase. Unit-tested.
- **`src/online/orchestrator.ts`** — the glue: connectivity/config check, lazy-load `firebase.ts`, anonymous sign-in, flush queue, cloud-save merge, expose higher-level ops (`submit`, `openLeaderboard`, `syncCloudSave`, `ensureName`, `rankTeaser`). Degrades to no-op-but-queue when offline/unconfigured.
- **`src/online/firebase.ts`** — the `OnlineBackend` implementation. **Only file that imports `firebase/*`.** Loaded via dynamic `import()` → its own Vite chunk.
- **`src/online/nickname.ts`** — **PURE**: `validateName(s)`, `suggestName(rng)` (deterministic given rng). Unit-tested.
- **`src/online/rank.ts`** — **PURE**: `formatRank(rank, total, lang)` using `Intl.NumberFormat`. Unit-tested.

### 4.3 OnlineBackend contract (the seam Firebase hides behind)

```ts
export type BoardId = 'global' | `d_${string}`;              // e.g. 'd_2026-07-04'
export interface ScoreRow { uid: string; name: string; score: number; at: number; }
export interface RankInfo { rank: number; total: number; }
export interface CloudSave { name: string; best: number; dailyBest?: { key: string; score: number }; updatedAt: number; }

export interface OnlineBackend {
  ready(): Promise<void>;                                   // init app + anonymous sign-in
  uid(): string | null;
  submitScore(board: BoardId, name: string, score: number): Promise<void>;
  topScores(board: BoardId, limit: number): Promise<ScoreRow[]>;
  myRank(board: BoardId, score: number): Promise<RankInfo>; // count(score > mine) + 1, plus total
  loadCloudSave(): Promise<CloudSave | null>;
  saveCloudSave(patch: Partial<CloudSave>): Promise<void>;
}
```

Tests inject a **fake in-memory backend** implementing this interface — no network, fully deterministic.

### 4.4 Graceful-degradation contract (the "managed well" guarantee)

- Online is enabled **iff** build-time config is present (`VITE_FIREBASE_PROJECT_ID`) **and** init succeeds. Otherwise every online op is a silent no-op and the game is byte-for-byte today's offline experience.
- Any online error (load failure, auth failure, rule rejection, timeout) is caught and swallowed; it **never** blocks play, never throws into the game loop, never shows a blocking error. Worst case: a score sits in the queue and a small "sync pending ⏳" hint shows.
- No online call is ever on the hot path of `step()` / `render()`.

### 4.5 Offline-first score flow

On game over (`main.ts`, after the existing local `saveBest`/`saveDailyBest`):

1. Compute `board`: `global` for free mode, `d_${runId.key}` for daily. (Daily also always contributes to `global`? **No** — daily runs post only to their daily board; free runs post to `global`. This matches the two existing modes and Game Center's two boards.)
2. Call `online.submit(board, score)`.
3. Orchestrator: if ready → `backend.submitScore`; on any failure → `queue.enqueue`. If not ready → `queue.enqueue`.
4. Queue flush happens on: successful `ready()`, `window` `online` event, and next app launch.

`queue.collapse()` keeps only the **max** score per board, so reconnecting after many offline runs posts one row per board, not a flood.

### 4.6 Cloud-save sync

On successful `ready()`:

1. `loadCloudSave()`. Merge with local `Save`: `best = max(local.best, remote.best)`; `dailyBest` = higher score for the matching `key`; `name` = remote name if present else local/suggested.
2. Write the merged result back (`saveCloudSave`) and to local `storage.ts`.
3. On each new local best thereafter, fire a `saveCloudSave({ best })` patch (best-effort, queued semantics not required — it's monotonic and idempotent).

Conflict resolution is deterministic: **max** for scores, last-write-wins (`updatedAt`) for name/settings. Unit-tested in `cloudsave.spec.ts`.

## 5. Firestore data model

```
users/{uid}
  { name: string, best: int, dailyBest?: { key: string, score: int }, updatedAt: int }

scores/{board}/entries/{uid}          // board ∈ { "global", "d_<dateKey>" }
  { name: string, score: int, at: int }
```

- **One row per uid per board** → boards are bounded by player count; no per-user spam.
- Top-N: `query(collection('scores', board, 'entries'), orderBy('score','desc'), limit(100))`.
- Rank: `getCountFromServer(query(..., where('score','>', mine)))` → `rank = count + 1`; `total = getCountFromServer(entries)`.
- Board keys derive from existing `dateKey()` in `src/core/daily.ts` (no new date math).

No composite index needed (single-field `score` ordering + single-field inequality). If the emulator/console requests one, commit `firestore.indexes.json`.

## 6. Backend integration

- **Firebase JS SDK (modular v10+)**, packages: `firebase/app`, `firebase/auth`, `firebase/firestore`. Added as an npm dependency but imported **only** in `src/online/firebase.ts` via dynamic import → isolated chunk.
- **Anonymous Auth**: `signInAnonymously`. The anonymous UID is the identity. (Google/Apple linking is Phase 3.)
- **Config via env**: `VITE_FIREBASE_API_KEY`, `VITE_FIREBASE_AUTH_DOMAIN`, `VITE_FIREBASE_PROJECT_ID`, `VITE_FIREBASE_APP_ID` (+ optional `STORAGE_BUCKET`, `MESSAGING_SENDER_ID`). Read via `import.meta.env`. These are **public by design** (client web config); the security boundary is Firestore Rules, not config secrecy. Set in `.env` (gitignored) and Vercel envs; committed `.env.example` documents the keys.

## 7. Security & soft anti-cheat (Rules-only, free Spark plan)

`firestore.rules` (committed, deployed via Firebase CLI):

```
rules_version = '2';
service cloud.firestore {
  match /databases/{db}/documents {
    function validName(n) { return n is string && n.size() >= 3 && n.size() <= 16; }
    function validScore(s) { return s is int && s >= 0 && s <= 100000; }   // generous cap; blocks absurd

    match /users/{uid} {
      allow read:  if request.auth != null && request.auth.uid == uid;
      allow write: if request.auth != null && request.auth.uid == uid
                   && validName(request.resource.data.name)
                   && validScore(request.resource.data.best)
                   && (!('best' in resource.data) || request.resource.data.best >= resource.data.best);
    }

    match /scores/{board}/entries/{uid} {
      allow read:   if true;                                  // public leaderboard + count aggregation
      allow create: if request.auth != null && request.auth.uid == uid
                    && validName(request.resource.data.name) && validScore(request.resource.data.score);
      allow update: if request.auth != null && request.auth.uid == uid
                    && validName(request.resource.data.name) && validScore(request.resource.data.score)
                    && request.resource.data.score > resource.data.score;   // monotonic — can only raise your own
    }
  }
}
```

Posture: casual soft anti-cheat. Rules enforce numeric type, a hard cap, monotonic raise, and name length. A determined user could still write any value ≤ cap from a console; **accepted** for a casual game. **Hardening upgrade path** (future, not Phase 1): move writes behind a callable Cloud Function that re-runs the seeded deterministic sim over a submitted input log to verify the score (the sim's purity makes this exact) — requires the Blaze plan.

## 8. UX & features

All new screens live in `src/ui/screens.ts` (or thin sibling modules it composes), styled in `src/styles.css`, strings in the `STRINGS` map.

- **Nickname (zero-friction):** first time the player would appear online (first online game-over, or opening the leaderboard), a modal shows a **pre-filled suggested name** (`suggestName`, e.g. "Zıpzıp#4821"). Accept or edit. Validated 3–16 chars, small profanity blocklist, uniqueness not required. Stored in `users/{uid}.name` + local. Editable in Settings.
- **Menu refresh:** existing Free/Daily buttons plus **"🏆 Sıralama"** (Leaderboard) and **"⚙ Ayarlar"** (Settings). Best line gains a rank teaser: *"Rekor 320 m · #128"* when known.
- **Leaderboard screen:** tabs **Global / Günlük**. Top-100 rows (rank · name · meters); the player's own row highlighted. If outside top-100, an "around me" slice (you ±2). Offline → cached rows + "çevrimdışı" badge + refresh. Skeleton while loading. Close → menu.
- **Game over:** online submit fires automatically (if online). New line: *"Sıralaman: #123 / 4.532 oyuncu"* (or *"senkron bekliyor ⏳"* when queued/offline). Buttons: Tekrar · Sıralamayı gör · Paylaş.
- **Settings screen (⚙):** sound toggle, haptics toggle (native only), language override (System/TR/EN, persisted), edit nickname, "Verileri sıfırla" (confirmed), version + credits. The quick 🔊 button stays.
- **Onboarding (minimal):** first-ever launch only — a one-line coach tip *"Sürükleyerek yönlendir 👆"* on first play, then fades. Gated by a `Save` flag. No multi-step tutorial in Phase 1.
- **i18n additions** to `STRINGS` tr/en: leaderboard, global, rank, players, settings, sound, haptics, language, system, nickname, enterName, invalidName, reset, confirm, cancel, offline, refresh, loading, syncPending, version, credits.

Rank/count numbers are formatted with `Intl.NumberFormat(lang)` (zero-dep, locale-correct) — no ambiguous "k/b/B" abbreviations.

## 9. Bundle budget & PWA/service-worker handling

- **Core entry chunk stays < 60KB gzip.** Firebase is a **separate dynamic-import chunk** and is excluded from the budget. Add a build/CI assertion that measures the core entry chunk's gzip size and fails if > 60KB (a small script over the Vite build output).
- **`vite.config.ts` `manualChunks`:** route `firebase` into its own chunk.
- **Workbox/PWA:** the Firebase chunk must **not** be precached (would bloat install and defeat lazy loading). Keep it out of the precache manifest (`globIgnores`/manifest filter) and **runtime-cache** it (`StaleWhileRevalidate`). `firestore.googleapis.com` / `identitytoolkit.googleapis.com` requests are runtime `NetworkFirst` (they need network anyway; failures fall through to the offline queue).

## 10. Testing strategy

Vitest, preserving the pure-core discipline. New **pure, unit-tested** units (no network, no DOM):

- `test/online/queue.spec.ts` — enqueue, collapse-to-max-per-board, drain ordering, quota/corruption fallback.
- `test/online/cloudsave.spec.ts` — merge: max(best), dailyBest by key, name LWW.
- `test/online/nickname.spec.ts` — `validateName` bounds + blocklist; `suggestName(rng)` deterministic.
- `test/online/rank.spec.ts` — `formatRank` across locales and magnitudes.
- `test/online/orchestrator.spec.ts` — submit/queue/flush/rank flows against the **fake in-memory backend**; asserts graceful degradation (backend throwing ⇒ enqueue, never propagates).

Existing sim/spawner/reachability tests are untouched and must stay green. `pnpm build` (tsc + vite) must pass.

## 11. Feature flag & rollout

- Online capability = `import.meta.env.VITE_FIREBASE_PROJECT_ID` present **and** init success. Absent ⇒ game ships exactly as today.
- One-time setup (implementation-time, owner does the login like the ASC flow; agent drives the CLI): create Firebase project, enable Anonymous Auth, create Firestore (production mode), register web app for config, `firebase deploy --only firestore:rules`.
- Ship to a Vercel **preview** first → verify airplane-mode + two-device leaderboard → promote to production.

## 12. iOS coexistence & follow-on

- The native Game Center bridge (`src/platform/capacitor.ts`) is **unchanged**. On iOS, game over fires **both** `bridge.submitScore` (Game Center) and `online.submit` (Firebase). No conflict; Game Center becomes the native bonus, Firebase the unified cross-platform board.
- Because Capacitor bundles the web assets, delivering this to the App Store requires a **new build + submission** — a small follow-on after the web ships and is validated. Firebase talks to HTTPS `*.googleapis.com`, so default ATS is fine; no plist allowlist needed.

## 13. Privacy & compliance (must-do for the iOS follow-on)

Phase 1 begins collecting a **chosen nickname + scores + an anonymous UID**. Therefore:

- Update `public/privacy.html` to disclose: anonymous identifier, user-provided nickname, and gameplay scores; stored in Firebase; no PII beyond the nickname the user types; purpose = leaderboard/save.
- The current ASC **"App Privacy: Data Not Collected"** declaration becomes **inaccurate** and must be updated (to "Data Collected: User Content (nickname), Identifiers (anonymous), Gameplay") before the iOS build ships. Tracked as a required item in the iOS follow-on, not a web blocker.

## 14. Roadmap context (later rounds — not this spec)

- **Phase 2 (v1.3):** achievements (offline-computed, cloud-synced), daily streak + Capacitor Local Notification reminder, rotating daily/weekly missions, profile screen.
- **Phase 3 (v1.4):** cosmetic character/theme unlocks (gated on milestones, no currency), friends + optional Google/Apple account linking for cross-device + friend boards.
- **Hardening (as needed):** Cloud Functions + seeded-replay score validation.

## 15. Acceptance criteria

1. Core entry chunk < 60KB gzip (CI-enforced); Firebase is a separate chunk.
2. With the network fully disabled (airplane mode) the game is fully playable; scores enqueue and sync on reconnect.
3. Online path: anonymous sign-in, submit, top-100 + rank render; a score set on one device appears on a second device's board.
4. Nickname flow works (suggest → accept/edit → persists); Settings screen functions; minimal onboarding shows once.
5. Graceful degradation verified: with config absent or Firebase forced to fail, the game behaves exactly as today.
6. All existing tests + the reachability invariant stay green; new pure units are tested; `pnpm build` passes.
7. No regression to free/daily modes or the native Game Center path.

## 16. Open decisions (resolved in brainstorm)

- Identity: **anonymous + nickname**, optional Google/Apple linking deferred to Phase 3. ✔
- Leaderboard: **unified Firebase everywhere**, Game Center kept as an iOS bonus. ✔
- Client integration: **Firebase JS SDK in a lazy chunk** (not hand-rolled REST). ✔
- Anti-cheat: **Rules-only on the free Spark plan**; Functions+replay is the future hardening path. ✔
- Phase 1 scope: **online foundation + unified leaderboard + cloud save + baseline UX** (achievements/streak/cosmetics deferred). ✔
